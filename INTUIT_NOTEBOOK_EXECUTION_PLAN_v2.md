### 1.4 — ⚠️ On the Absence of a Merchant Identifier

> **Critical design decision — must be narrated clearly in a markdown cell.**

The dataset has no merchant ID field. It has accounts. These are not the same thing.

A real merchant may have opened **multiple accounts** under different dates, especially if they're cycling accounts to avoid detection (which is itself a fraud pattern). Any attempt to construct a synthetic merchant ID using `account_open_date` as a component would collapse multi-account behavior — the exact signal we want to detect — into false uniqueness.

**The correct approach:** Work entirely at the **account level** throughout the notebook. Use the term "account" not "merchant" in all code, comments, and charts. For clustering and anomaly detection (Section 4), the unit of analysis is one account (aggregated across its transactions), not one merchant.

**For segment-level analysis** (e.g., "which channel is riskiest"), use **segment aggregates** — group by channel, MCC, state, credit tier — not by any account-level composite key pretending to be a merchant.

**Callout for the presentation:** Explicitly flag this in the methodology slide: *"Without a persistent merchant identifier, we cannot track behavior across accounts opened by the same entity. This is a known limitation and one of the first things we'd fix with real production data — cross-account linking using shared attributes like phone, bank account, device fingerprint, or IP is a standard fraud detection capability."*

This limitation, stated honestly, demonstrates analytical maturity. Hiding it and pressing forward with a flawed proxy is the opposite.

### 1.5 — Account-Level Aggregation Table

Aggregate transactions by a true account-level key (constructed from the stable account attributes — everything except `account_open_date` is fine as an identifier component if it doesn't change per-account. In practice, treat each unique combination of `channel + account_open_date + location_state + mcc` as one account entity, acknowledging that if the same real-world merchant opened two accounts on different dates, they appear as two separate accounts here — which is fine and actually correct at the account level).

Per-account aggregates:
- Total txn volume ($), txn count
- Chargeback rate (events), IntuitLoss rate ($)
- Avg and max txn amount
- Account age at first dispute (if any)
- Txn velocity: txn count / account_age_days
- First txn date relative to open date (days-to-first-transaction)
- Whether account is open at end of period
- Close reason (if closed)

## ⚠️ Critical Methodological Notes (Embedded as Markdown Cells in Notebook)

These must appear as standalone markdown cells immediately before the relevant code section — they become Q&A talking points:

1. **On no merchant identifier:** *"This dataset tracks accounts, not merchants. A single merchant could open multiple accounts — especially a fraudulent one, which is literally the behavior we're trying to detect. Collapsing on account_open_date would destroy this signal. We work at the account level throughout and flag this as a data infrastructure gap."*

2. **On temporal train/test split:** *"Fraud models trained on randomly shuffled data are worse than useless — they're confidently wrong. Future data leaks into the training set and inflates every metric. We always split on time."*

3. **On SMOTE placement:** *"SMOTE only touches the training set. Oversampling before splitting is one of the most common mistakes in imbalanced classification — it causes data leakage by creating synthetic copies of test-set points."*

4. **On leakage in segment risk scores:** *"Channel loss rate, MCC loss rate, and state loss rate are computed on the full dataset for EDA purposes, but recomputed exclusively on training folds for the supervised model. They're computed with a temporal lag in any real deployment."*

5. **On forecasting with 12 data points:** *"Twelve months is a brutally short time series. SARIMA and Prophet are being asked to extrapolate from a single calendar year with no seasonal validation. The ensemble and Monte Carlo approaches are how we stay honest about that uncertainty — we're not pretending to precision we don't have."*

6. **On 2021 external events:** *"The spikes we see in the data don't live in a vacuum. Each anomaly in the residuals can be cross-referenced against the macroeconomic and fraud event calendar. This matters for forecasting — if we believe 2022 will have a similar stimulus disbursement pattern (e.g., extended CTC payments, tax refund season), we can incorporate that as a regressor in Prophet rather than letting it show up as unexplained noise."*

7. **On SHAP vs. raw feature importance:** *"Tree-based feature importance is biased toward high-cardinality features and splits. SHAP values are theoretically grounded (Shapley values from cooperative game theory), model-agnostic, and give directional effect. They're the right tool for explaining a fraud model to non-technical stakeholders."*

8. **On the KM censoring assumption:** *"Kaplan-Meier assumes non-informative censoring — that accounts still open at Dec 31 aren't systematically more or less likely to chargeback than closed accounts. In fraud data this is probably violated: accounts closed for cause are more likely to have chargebacks. Worth flagging as a limitation of the survival analysis."*
