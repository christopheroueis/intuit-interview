## 🧹 Section 1 — Data Loading, Cleaning & Feature Engineering

### 1.1 — Load & Parse

Parse all date columns as datetime. **Use `utc=True` for `account_open_date`, `close_date`, and `cb_date`** (they have timezone offsets), then strip timezone for arithmetic. `txn_date` is tz-naive. Standardize string columns (strip, lowercase). Print null-rate report.

> [!WARNING]
> **v3 fix:** The date columns `account_open_date`, `close_date`, `cb_date` contain timezone-aware timestamps (e.g., `2015-06-11T05:28:37.000-07:00`), while `txn_date` is timezone-naive (`2021-01-26`). Mixing these causes `TypeError: Cannot subtract tz-naive and tz-aware datetime-like objects`. Parse with `utc=True` then `.dt.tz_localize(None)`.

### 1.2 — Outcome Encoding

```python
df["is_disputed"]    = df["cb_date"].notna()
df["is_intuit_loss"] = df["outcome"] == "IntuitLoss"
df["is_merch_loss"]  = df["outcome"] == "MerchLoss"
df["is_merch_win"]   = df["outcome"] == "MerchWin"
df["is_reversal"]    = df["outcome"] == "Reversal"
```

### 1.3 — Feature Engineering (Transaction Level)

| Feature | Formula | Notes |
|---|---|---|
| `account_age_days` | `txn_date − account_open_date` | Core fraud signal |
| `is_new_account` | `account_age_days < 30` | Boolean flag |
| `is_very_new_account` | `account_age_days < 7` | Tighter flag for bust-out detection |
| `txn_month` | month integer | Temporal feature |
| `txn_quarter` | quarter integer | Temporal feature |
| `days_to_dispute` | `cb_date − txn_date` | Null if no dispute |
| `account_lifespan_days` | `close_date − account_open_date` | Null if still open |
| `chargeback_ratio` | `chargeback_amt / txn_amount` | Capped at 1.0 |

**Segment-level risk scores** (computed on full dataset for EDA; recomputed on train-only for ML in Section 6):
- `channel_loss_rate`: IntuitLoss $ / total txn $ by channel
- `mcc_loss_rate`: IntuitLoss $ / total txn $ by MCC
- `state_loss_rate`: IntuitLoss $ / total txn $ by state

### 1.4 — ⚠️ On the Absence of a Merchant Identifier

> **Critical design decision — must be narrated clearly in a markdown cell.**

The dataset has no merchant ID field. It has accounts. A real merchant may have opened **multiple accounts** under different dates, especially if cycling accounts to avoid detection.

**The correct approach:** Work entirely at the **account level**. Use the term "account" not "merchant" in all code, comments, and charts.

**Callout for the presentation:** *"Without a persistent merchant identifier, we cannot track behavior across accounts opened by the same entity. This is a known limitation — cross-account linking using shared attributes like phone, bank account, or device fingerprint is a standard fraud detection capability we'd recommend."*

### 1.5 — Account-Level Aggregation Table

Treat each unique combination of `channel + account_open_date + location_state + mcc` as one account entity.

> [!IMPORTANT]
> **Actual data:** This produces ~189,548 unique accounts. Of these, **81 had at least one IntuitLoss** (0.043% prevalence). This extreme imbalance drives every downstream modeling decision.

Per-account aggregates:
- Total txn volume ($), txn count
- Chargeback rate (events), IntuitLoss rate ($)
- Avg and max txn amount
- Account age at first dispute (if any)
- Txn velocity: txn count / account_age_days
- First txn date relative to open date
- Whether account is open at end of period
- Close reason (if closed)

Print class imbalance report.

## 📈 Section 7 — Time-Series Forecasting: 2022 Loss Outlook

> [!WARNING]
> **v3 fix — fundamental restructuring of this section:**
>
> The v2 plan used 12 monthly data points for SARIMA, Prophet, Holt-Winters, and STL decomposition. Critical issues:
>
> 1. **STL decomposition** with `period=12` on 12 points is **mathematically impossible** — the seasonal component requires at least 2 full cycles (24 points). **Removed from main analysis; replaced with a simple trend + moving average decomposition.**
> 2. **Holt-Winters additive seasonal** requires `2 × period` minimum observations. With `period=12`, that's 24 points. WILL CRASH. **Fix: Use Holt-Winters with `seasonal=None` (trend-only exponential smoothing).**
> 3. **SARIMA** with `auto_arima`: will run but **cannot estimate seasonal parameters** from a single calendar cycle. Suppress seasonal component (`seasonal=False` or let auto_arima select `m=1`). Document this explicitly.
> 4. **Prophet**: can technically handle 12 points but confidence intervals will be extremely wide and results unreliable. Keep it, but frame honestly.
>
> **Alternative considered but rejected:** Weekly aggregation gives ~52 data points (better statistically), but the assignment specifically asks for **monthly loss predictions for Jan–Dec 2022**, so the forecasting target must be monthly. We forecast monthly, but acknowledge the statistical limitations.
>
> **Bottom line:** The honest story is that 12 monthly observations is not enough for reliable seasonal time-series forecasting. The ensemble + Monte Carlo approach is how we communicate uncertainty. The presentation should frame these as *scenario estimates*, not precise predictions.

## ⚠️ Critical Methodological Notes (Embedded as Markdown Cells)

1. **On no merchant identifier:** *"This dataset tracks accounts, not merchants. A single merchant could open multiple accounts — especially a fraudulent one. We work at the account level throughout and flag this as a data infrastructure gap."*

2. **On temporal train/test split:** *"Fraud models trained on randomly shuffled data are worse than useless — they're confidently wrong. We split on time. Our split uses first_txn_date, placing accounts first seen in Oct–Dec 2021 into the test set."*

3. **On SMOTE placement:** *"SMOTE only touches the training set. Oversampling before splitting causes data leakage by creating synthetic copies of test-set points."*

4. **On leakage in segment risk scores:** *"Channel/MCC/state loss rates are computed on the full dataset for EDA, but recomputed exclusively on training data for the supervised model."*

5. **On forecasting with 12 data points:** *"Twelve months is a brutally short time series. We cannot estimate seasonality from a single calendar cycle. The ensemble and Monte Carlo approaches are how we stay honest about uncertainty."*

6. **On 2021 external events:** *"The spikes don't live in a vacuum. Each anomaly can be cross-referenced against the macro/fraud event calendar. This matters for forecasting — if 2022 has similar dynamics, we can incorporate them as regressors."*

7. **On SHAP vs. raw feature importance:** *"Tree-based feature importance is biased toward high-cardinality features. SHAP values are theoretically grounded and give directional effect — the right tool for explaining a fraud model to non-technical stakeholders."*

8. **On the KM censoring assumption:** *"Kaplan-Meier assumes non-informative censoring. In fraud data this is probably violated: accounts closed for cause are more likely to have chargebacks. Worth flagging as a limitation."*
